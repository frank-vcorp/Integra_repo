### 1️⃣ Server Actions: `clinical-history.actions.ts`

**Ubicación:** `frontend/src/actions/clinical-history.actions.ts`

#### Funciones Implementadas:

**`getWorkerClinicalHistory(workerId: string)`**
- Obtiene el historial clínico persistente de un trabajador
- Retorna: `{ success: boolean, data?: ClinicalHistory, error?: string }`
- Validaciones: ID de trabajador obligatorio
- Include: Datos de trabajador (nombre, universal ID)

**`upsertWorkerClinicalHistory(workerId: string, data: any)`**
- Crea o actualiza el historial clínico de forma atómica
- Validación Zod contra `ClinicalHistoryDataSchema`
- Comportamiento: UPSERT (insert si no existe, update si existe)
- Revalidación automática de rutas: `/history/{workerId}` y `/workers/{workerId}`

## 🗄️ Mapeo de Base de Datos

### Modelo Prisma: `ClinicalHistory`

```prisma
model ClinicalHistory {
  id          String   @id @default(uuid())
  workerId    String   @unique
  worker      Worker   @relation(fields: [workerId], references: [id])
  
  // Schema-Driven Storage
  data        Json     @default("{}")
  
  lastUpdated DateTime @default(now())
  updatedAt   DateTime @updatedAt
  
  @@map("clinical_histories")
}
```

## 📝 Notas Técnicas

### Decisiones de Diseño:

**1. JSON vs Tabla Normalizada**
✅ Se mantiene JSON per SPEC INTEGRA
- Flexibilidad para future modules
- Fácil versionado de estructura
- Evita denormalización excesiva

**2. UPSERT atómico**
✅ Prisma `.upsert()` con `@unique` en workerId
- Previene race conditions
- Una sola query a BD
- Transacción implícita

**3. Estructura de Tabs**
✅ Cliente-side state management (React)
- Evita round-trips innecesarios
- UX más fluida
- Validación centralizada en submit
