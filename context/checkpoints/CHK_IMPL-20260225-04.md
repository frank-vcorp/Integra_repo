### 3. **Nuevo Modelo: `AuditLog`**
**Ubicación:** Línea ~301 en `frontend/prisma/schema.prisma`

**Campos:**
- `id` (String, PK, uuid)
- `userId` (String?, FK → User, opcional)
  - Por si la acción la realiza el sistema (sin usuario)
- `action` (String) - Tipo de acción: CREATE, UPDATE, DELETE, LOGIN, etc.
- `entity` (String) - Nombre de la entidad: Worker, Appointment, MedicalEvent, etc.
- `entityId` (String?, opcional) - ID de la entidad afectada
- `details` (Json?, opcional) - Datos adicionales (ej. valores antes/después)
- `ipAddress` (String?, opcional) - Dirección IP del cliente
- `createdAt` (DateTime, default: now())

**Tabla:** `audit_logs`

**Relaciones:**
- N:1 opcional con User (múltiples logs pueden estar asociados a un usuario, o ninguno)

## 🎯 Decisiones de Diseño

1. **AppointmentStatus enum:** Valores claros y exhaustivos (SCHEDULED, CONFIRMED, CANCELLED, NO_SHOW, COMPLETED) para cobertura total del ciclo de vida de una cita.

2. **Relación 1:1 opcional (Appointment → MedicalEvent):**
   - Una cita puede no generar un evento inmediatamente
   - Flexibilidad para citas que se cancelan antes de check-in
   - Un evento médico existe independientemente (para servicios no asociados a cita)

3. **userId opcional en AuditLog:**
   - Permite registros de acciones del sistema o procesos automáticos
   - Mantiene trazabilidad sin requerir usuario específico

4. **details como Json en AuditLog:**
   - Flexibilidad máxima para registrar cambios específicos de cualquier entidad
   - Soporta antes/después, cambios parciales, contexto adicional
